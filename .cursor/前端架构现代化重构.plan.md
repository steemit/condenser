# 前端架构现代化重构计划

## 项目现状分析

当前项目使用：

- React 15.6.2 + React Router 3.2.0
- Redux 3.7.2 + Redux-Saga 0.16.0 + Immutable.js
- Webpack 3.7.1 + Babel 6
- Koa 1.4.1 (自定义 SSR)
- SCSS + Foundation Sites

## 目标架构

- **React 19** (最新版本)
- **Next.js 15** (App Router，内置 SSR)
- **Redux Toolkit** (现代化 Redux)
- **TypeScript** (逐步迁移)
- **Tailwind CSS** (可选，替代 Foundation)

## 重构步骤

### 阶段一：项目基础设施搭建

1. **初始化 Next.js 项目**

- 在项目根目录创建 `nextjs/` 目录
- 使用 `create-next-app@latest` 初始化，选择 App Router
- 配置 TypeScript 支持
- 迁移现有配置文件（`config/` 目录）

2. **依赖管理迁移**

- 分析 `package.json` 中的依赖
- 升级兼容的依赖包
- 替换废弃的包（如 `react-router-redux` → Next.js 路由）
- 移除不再需要的包（如 `webpack-isomorphic-tools`, `iso`）

3. **构建配置q**

- 配置环境变量（`.env.local`, `.env.production`）
- 设置路径别名（`tsconfig.json` 或 `next.config.js`）

### 阶段二：状态管理迁移

4. **Redux Toolkit 迁移**

- 安装 `@reduxjs/toolkit` 和 `react-redux`
- 创建新的 store 结构（`store/index.ts`）
- 迁移 Reducers：
- `src/app/redux/AppReducer.js` → `store/slices/appSlice.ts`
- `src/app/redux/GlobalReducer.js` → `store/slices/globalSlice.ts`
- `src/app/redux/UserReducer.js` → `store/slices/userSlice.ts`
- 其他 reducers 类似迁移
- 移除 Immutable.js，使用原生 JavaScript 对象
- 使用 RTK Query 替代部分 Redux-Saga（可选）

5. **Redux-Saga 迁移**

- 评估每个 Saga 的功能
- 优先使用 RTK Query 替代数据获取相关的 Saga
- 保留复杂业务逻辑的 Saga，迁移到 RTK 兼容版本
- 迁移文件：`src/app/redux/*Saga.js` → `store/sagas/*.ts`

### 阶段三：路由和页面迁移

6. **路由系统迁移**

- 分析 `src/app/RootRoute.js` 和 `src/app/ResolveRoute.js`
- 将 React Router 3 路由转换为 Next.js App Router
- 创建对应的路由结构：
- `app/` 目录（App Router）
- `app/page.tsx` (首页)
- `app/@[username]/page.tsx` (用户页面)
- `app/[category]/@[username]/[permlink]/page.tsx` (文章页面)
- 其他页面类似迁移

7. **页面组件迁移**

- 迁移页面组件：`src/app/components/pages/` → `app/**/page.tsx`
- 更新组件导入路径
- 使用 Next.js 的 `use client` 和 `use server` 指令
- 迁移服务端数据获取逻辑到 Server Components 或 Route Handlers

### 阶段四：组件迁移

8. **共享组件迁移**

- 迁移 `src/app/components/elements/` → `components/elements/`
- 迁移 `src/app/components/cards/` → `components/cards/`
- 迁移 `src/app/components/modules/` → `components/modules/`
- 更新组件为函数式组件（如果还是类组件）
- 添加 TypeScript 类型定义

9. **样式系统迁移**

- 评估是否迁移到 Tailwind CSS
- 如果保留 SCSS，配置 Next.js 支持 SCSS
- 迁移样式文件到对应组件目录
- 更新 Foundation Sites 类名为现代 CSS 或 Tailwind

### 阶段五：SSR 和 API 迁移

10. **SSR 功能迁移**

- Next.js App Router 内置 SSR，移除自定义 SSR 代码
- 迁移 `src/server/app_render.jsx` 逻辑到 Next.js Server Components
- 迁移 `src/shared/UniversalRender.jsx` 的数据获取逻辑
- 使用 Next.js 的 `generateMetadata` 处理 SEO

11. **API 路由迁移**

- 迁移 `src/server/api/` → `app/api/` (Route Handlers)
- 迁移 `src/server/json/` → `app/api/`
- 更新 API 调用方式（使用 Next.js 的 fetch 或 Route Handlers）

12. **服务端中间件**

- 评估 Koa 中间件功能
- 迁移必要的中间件到 Next.js Middleware (`middleware.ts`)
- 处理会话、CSRF、重定向等逻辑

### 阶段六：工具和工具函数迁移

13. **工具函数迁移**

- 迁移 `src/app/utils/` → `lib/utils/` 或 `utils/`
- 迁移 `src/shared/` → `lib/shared/`
- 添加 TypeScript 类型定义

14. **国际化迁移**

- 评估 `react-intl` 的使用
- 考虑迁移到 `next-intl` 或保持 `react-intl`
- 迁移翻译文件：`src/app/locales/` → `locales/` 或 `messages/`

### 阶段七：测试和优化

15. **测试基础设施搭建**

- 更新 Jest 配置以支持 Next.js 和 TypeScript
- 配置 React Testing Library
- 配置测试覆盖率工具（如 Istanbul）
- 设置测试环境变量和 Mock 数据

16. **功能函数单元测试**

- 为核心功能函数编写单元测试
- 测试工具函数：`lib/utils/` 下的所有工具函数
- 测试 Redux Reducers 和 Actions
- 测试 API 调用函数
- 测试业务逻辑函数
- **目标覆盖率**：功能函数测试覆盖率达到 80% 以上

17. **UI 组件单元测试（重要）**

- **所有 UI 组件都需要有单元测试**
- 使用 React Testing Library 测试组件渲染
- 测试组件的用户交互（点击、输入、选择等）
- 测试组件的 Props 变化和状态更新
- 测试组件的条件渲染和错误处理
- 测试组件的可访问性（Accessibility）
- **组件测试覆盖范围**：
  - `components/elements/` 下的所有组件
  - `components/cards/` 下的所有组件
  - `components/modules/` 下的所有组件
  - 页面组件（`app/**/page.tsx`）的关键交互
- **目标覆盖率**：UI 组件测试覆盖率达到 70% 以上

18. **集成测试**

- 测试页面级别的功能集成
- 测试 Redux Store 与组件的集成
- 测试 API 调用与组件的集成
- 测试路由导航

19. **测试最佳实践**

- 使用描述性的测试名称
- 遵循 AAA 模式（Arrange, Act, Assert）
- 避免测试实现细节，关注用户行为
- 使用 Mock 和 Spy 隔离依赖
- 保持测试独立性和可重复性
- 测试文件命名：`*.test.tsx` 或 `*.spec.tsx`

20. **性能优化**

- 使用 Next.js Image 组件优化图片
- 实现代码分割和懒加载
- 优化 Bundle 大小
- 配置缓存策略

21. **开发体验优化**

- 配置 ESLint 和 Prettier
- 设置 Git hooks（pre-commit 运行测试）
- 配置 Storybook（用于组件开发和测试）
- 更新开发脚本
- 配置测试覆盖率报告

### 阶段八：部署和文档

18. **部署配置**

- 更新 Dockerfile 以支持 Next.js
- 配置生产环境变量
- 设置 CI/CD 流程
- 性能监控和错误追踪

19. **文档更新**

- 更新 README.md
- 编写迁移指南
- 更新贡献指南
- API 文档（如果需要）

## 关键文件和目录

### 需要重点迁移的文件：

- `src/app/Main.js` → Next.js 入口（App Router 自动处理）
- `src/app/RootRoute.js` → `app/` 目录结构
- `src/shared/UniversalRender.jsx` → Server Components + Route Handlers
- `src/server/server.js` → Next.js 服务器（内置）
- `src/app/redux/` → `store/` (Redux Toolkit)
- `webpack/` → 移除（Next.js 内置构建）

### 需要保留的业务逻辑：

- Steem API 集成 (`@steemit/steem-js`)
- 区块链相关功能
- 用户认证和会话管理
- 内容管理系统

## 迁移策略

1. **Git 分支管理**

   - 在新分支 `refactor/nextjs-migration` 上进行所有工作
   - 每个阶段完成后提交，保持清晰的提交历史
   - 重要里程碑创建标签（tag）便于回滚

2. **目录结构**

   - 立即将旧代码移动到 `legacy/` 目录，保持根目录整洁
   - 根目录直接使用 Next.js 项目结构

3. **核心功能优先**

   - 优先迁移核心功能：用户认证和登录系统
   - 其他功能后续逐步迁移
   - 确保核心功能完整可用后再迁移其他模块

4. **API 重新设计**

   - 不需要保持与旧系统的 API 兼容性
   - 可以重新设计更现代的 API 结构
   - 使用 Next.js Route Handlers 实现 API

5. **测试策略**

   - 重写测试以适配新架构
   - 使用 React Testing Library 替代 Enzyme
   - 为核心功能编写完整的测试覆盖

6. **功能对等**

   - 确保每个功能在新系统中都有对应实现
   - 建立功能对比清单，逐一验证

7. **性能对比**

   - 迁移后对比性能指标
   - 记录性能改进数据

## 风险评估

- **高风险**：SSR 逻辑迁移、路由系统重构
- **中风险**：Redux 状态管理迁移、组件迁移
- **低风险**：工具函数迁移、样式迁移

## Git 工作流

### 分支策略

```
main/master (主分支，保持稳定)
  └── refactor/nextjs-migration (重构分支)
       ├── feature/migrate-redux (子功能分支，可选)
       ├── feature/migrate-routes (子功能分支，可选)
       └── ... (其他功能分支)
```

### 提交规范

- 使用清晰的提交信息，格式：`[阶段] 简短描述`
- 示例：`[阶段一] 初始化 Next.js 项目结构`
- 示例：`[阶段二] 迁移 AppReducer 到 Redux Toolkit`

### 里程碑标记

- 每个阶段完成后创建标签
- 格式：`refactor/v1.0-phase-1`, `refactor/v1.0-phase-2` 等
